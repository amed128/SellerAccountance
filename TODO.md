# Backlog

Left-aside items, roughly in priority order.

- **Password reset**: no way to recover an account after a forgotten password — a hard lockout for a real user. Needs a transactional email service (e.g. [Resend](https://resend.com), free tier). Blocked on picking a provider.
- **E-mail verification on signup**: less urgent than password reset (doesn't lock anyone out), but should land before a real public launch. Depends on the same email service.
- **Isolated preview database branches**: Preview deployments currently share the same Neon database as Production (chose the simple option when connecting the integration). Revisit enabling per-deployment Neon branches for Preview once things stabilize, so PR previews don't touch production data.
- **International expansion, phase 2 — UK**: EU home-country generalization (any EU country, not just France) shipped. UK is next — still VAT, but outside the EU OSS system post-Brexit (separate registration/rates/thresholds, files with HMRC not DGFIP).
- **International expansion, phase 3 — US**: not VAT — destination-based sales tax, state-by-state, with Amazon acting as marketplace facilitator in most states. A genuinely different tax engine, not an extension of `vat.ts`.
- **International expansion, phase 4 — Middle East / Asia**: each country is its own VAT/GST system (UAE, Japan, India's GST, etc.) — build out per-country as demand justifies it.
- **Per-country franchise/exemption wording**: the franchise-en-base exemption note is currently generic (no legal article citation), since that varies per EU country. Worth adding precise per-country citations once there are non-French users actually relying on it.
- **Real gross margin (COGS)**: fees reverse charge, sourcing invoice entry, and regime-aware deduction are all done (see below) — `vatToPay` on the overview page now correctly nets out deductible sourcing VAT. The one piece left is cost tracking: the app still doesn't derive COGS from sourcing invoices, so sellers see revenue and VAT correctly but not real profit margin. Likely 1 more PR:
  - ~~Amazon fees reverse charge~~ — done: `feesReverseChargeVatDue`/`feesReverseChargeVatDeductible` in `vat.ts`. STANDARD-regime sellers see it net to zero in `vatToPay` (due and deductible both shown, for the return); FRANCHISE-regime sellers correctly owe it as a real cost, since franchise en base blocks deduction but not the reverse-charge self-assessment obligation itself — a nuance the original scoping note here missed.
  - ~~Sourcing invoice entry~~ — done: `SourcingInvoice` model + `/sourcing` manual entry form (supplier, date, SKU, quantity, HT/VAT/TTC amounts exactly as shown on the invoice — not recomputed from a rate, currency, and VAT treatment: domestic / intra-EU reverse charge / import, with a UI hint that import VAT deduction needs the customs document, not just this invoice). CSV import still not built.
  - ~~Regime-aware deduction~~ — done: `computeVatSummary`/`monthlySummaries` now take sourcing invoices and add `sourcingDeductibleVat`/`sourcingNonDeductibleVat` to `VatSummary`, netted into `vatToPay` on the overview page (not per-report pages, since invoices aren't tied to a specific Amazon export). `FRANCHISE` never deducts; `REVERSE_CHARGE` is self-assessed at the home rate like Amazon fees (due = deductible, net zero for STANDARD); `IMPORT` is deliberately withheld from deduction for everyone pending a way to track the customs clearance document, rather than silently overstating what's reclaimable.
  - **Unit cost / COGS integration**: derive per-SKU HT unit cost from sourcing invoices (weighted-average across POs — simplest for this scale, no need for true FIFO lot-tracking), then extend period summaries with `cogs`/`grossMargin` so revenue reporting reflects actual profit, not just Amazon payout.
