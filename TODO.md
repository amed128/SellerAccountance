# Backlog

Left-aside items, roughly in priority order.

- **Password reset**: no way to recover an account after a forgotten password — a hard lockout for a real user. Needs a transactional email service (e.g. [Resend](https://resend.com), free tier). Blocked on picking a provider.
- **E-mail verification on signup**: less urgent than password reset (doesn't lock anyone out), but should land before a real public launch. Depends on the same email service.
- **Isolated preview database branches**: Preview deployments currently share the same Neon database as Production (chose the simple option when connecting the integration). Revisit enabling per-deployment Neon branches for Preview once things stabilize, so PR previews don't touch production data.
- **International expansion, phase 2 — UK**: EU home-country generalization (any EU country, not just France) shipped. UK is next — still VAT, but outside the EU OSS system post-Brexit (separate registration/rates/thresholds, files with HMRC not DGFIP).
- **International expansion, phase 3 — US**: not VAT — destination-based sales tax, state-by-state, with Amazon acting as marketplace facilitator in most states. A genuinely different tax engine, not an extension of `vat.ts`.
- **International expansion, phase 4 — Middle East / Asia**: each country is its own VAT/GST system (UAE, Japan, India's GST, etc.) — build out per-country as demand justifies it.
- **Per-country franchise/exemption wording**: the franchise-en-base exemption note is currently generic (no legal article citation), since that varies per EU country. Worth adding precise per-country citations once there are non-French users actually relying on it.
- ~~Deductible input VAT on sourcing, and real gross margin~~ — **all 4 slices done**, code-complete:
  - ~~Amazon fees reverse charge~~ — `feesReverseChargeVatDue`/`feesReverseChargeVatDeductible` in `vat.ts`. STANDARD-regime sellers see it net to zero in `vatToPay` (due and deductible both shown, for the return); FRANCHISE-regime sellers correctly owe it as a real cost, since franchise en base blocks deduction but not the reverse-charge self-assessment obligation itself.
  - ~~Sourcing invoice entry~~ — `SourcingInvoice` model + `/sourcing` manual entry form (supplier, date, SKU, quantity, HT/VAT/TTC amounts exactly as shown on the invoice, currency, and VAT treatment: domestic / intra-EU reverse charge / import). CSV import still not built — a candidate for a later, separate item if wanted.
  - ~~Regime-aware deduction~~ — `sourcingDeductibleVat`/`sourcingNonDeductibleVat` in `VatSummary`, netted into `vatToPay` on the overview page. `FRANCHISE` never deducts; `REVERSE_CHARGE` self-assessed at the home rate like Amazon fees; `IMPORT` withheld from deduction for everyone pending a way to track the customs clearance document.
  - ~~Unit cost / COGS integration~~ — `computeUnitCosts()` derives a weighted-average HT unit cost per SKU from the *full* invoice history (not period-scoped, since a unit's cost basis is cumulative — unlike VAT deduction timing). `cogs`/`grossMargin` added to `VatSummary`, shown on the overview page (aggregate + monthly breakdown) once at least one sourcing invoice exists; a `cogsIncomplete` note flags when some sold SKUs have no matching invoice, so margin is understated in coverage rather than silently wrong.

Automated tests (70 unit + 20 e2e, all passing) cover the math and basic rendering, but haven't been eyeballed end-to-end by a human yet.
